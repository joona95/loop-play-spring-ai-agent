## 1단계: ChatMemory 3레이어 + X-Session-Id + 지시 대명사 시나리오

**목표**: Memory 3레이어를 직접 조립하고, `X-Session-Id`로 고객별 세션 분리가 동작하는지 5종 시나리오로 검증한다.

### 구현 개요

### 핵심 Finding

1. **세션 분리는 완벽하게 동작한다 (시나리오 4)** — `X-Session-Id` → `.param(ChatMemory.CONVERSATION_ID, …)` 경로로 conversationId가 호출 단위로 주입되어, 세션 간 메모리가 *전혀* 섞이지 않는다. 시나리오 4의 B 세션이 A의 `1234`를 전혀 모르고, `/session/ids`에 세션이 독립 키로 등록됨으로 확인.

2. **지시 대명사("그거"/"그 주문")는 메모리로 정확히 해소된다 (시나리오 1·3)** — 모델이 이전 턴의 `2024-1234`를 끌어와 지시 대명사를 해소. *맥락 유지*라는 메모리의 핵심 목적은 달성.

3. **메모리 재활용 시 Tool을 재호출하지 않고 *stale 데이터*를 재서술한다 (시나리오 1)** — "그거 언제 도착?"에 `getDeliveryStatus`를 다시 부르지 않고 *이전 답을 메모리에서 재서술* → 도착 연도가 **2026→2024로 드리프트**. 메모리는 토큰을 아끼지만 *실시간성*을 잃는다. **운영 함의**: 배달 상태처럼 *시시각각 변하는* 데이터는 메모리 재활용이 아니라 Tool 재호출을 강제해야 한다(예: 시스템 프롬프트에 "배달 상태는 매번 getDeliveryStatus로 재확인" 룰).

4. **메모리는 *대상 전환*도 이해한다 (시나리오 2)** — "그거 말고 1235"에서 2회차가 1235를 지칭. 단 cancelOrder 자체는 미호출(Round 2의 promise hallucination 재현) — 이는 메모리와 무관한 *tool-calling 능력 천장* 문제.

5. **삭제는 즉시·완전하다 (시나리오 5)** — `clear` 후 `/messages`가 `[]`, 후속 지시 대명사가 맥락을 잃음. 감사 로그까지 정상.

6. **언어 드리프트는 메모리와 무관하게 간헐 재현된다 (Round 2 finding 재현)** — 3회 반복 로그에서 `라이der` 10건 · `라이ダー` 4건 · 정상 `배달원` 2건, 그리고 **시나리오 5 R1은 중국어로 폭주**(*"현재 라이ダー는 배달 시작阶段，请继续用韩语回复…"* + 응답 포맷까지 누출). 메모리 도입이 드리프트를 *유발하지도 완화하지도* 않음 — Round 2의 *토큰 단위 다국어 오염* 가설이 메모리 컨텍스트에서도 그대로 관측. **메모리 레이어와 직교하는 문제**임을 3회 반복이 분리 확인.

> **재현율 메타 교훈**: 메모리 5종은 3/3, Tool은 0/3, 드리프트는 간헐 — *같은 코드라도 어느 레이어는 결정론적이고 어느 레이어는 확률적*이다. 한 번 측정으로 "동작한다/안 한다"를 단정하지 말 것(Round 2 §1 교훈의 재확인).

### Observability 함정 → 해결 — DEBUG 로그에 메모리를 *보이게* 만들기

**증상(수정 전)**: `SimpleLoggerAdvisor`가 찍는 `request:` 로그의 messages 배열이 **`[SystemMessage, 현재 UserMessage]` 뿐**이라 주입된 이전 대화가 안 보였다. 메모리가 동작 안 한 게 아니라 *관측 지점*이 잘못된 것:

- advisor 실행 순서는 order 오름차순(낮을수록 바깥=먼저). 기본 `SimpleLoggerAdvisor`(order 0) **<** `MessageChatMemoryAdvisor.order(10)`
- → **로깅이 메모리 주입 *이전*에 실행**되어, 로그엔 주입 전 프롬프트가 찍혔다.

**해결**: `SimpleLoggerAdvisor`의 order를 **20**(memory의 10보다 큼 = 주입 *후* 실행)으로 올렸다 — `ChatClientConfig.simpleLoggerAdvisor()`에서 `new SimpleLoggerAdvisor(20)`.

**실측 증거** (local 프로파일, 세션 `debugfix1`, → raw [`debug-memory-injection.txt`](raw/debug-memory-injection.txt)):

| 턴 | request 로그의 messages 구성 |
|---|---|
| 턴1 "2024-1234 어디쯤?" | `System=1, User=1, Assistant=0` (주입할 이력 없음) |
| 턴2 "그거 언제 도착?" | `System=1, User=2, Assistant=1` — **턴1 USER+ASSISTANT가 주입돼 로그에 보임** |

턴2 request 로그에 턴1의 `UserMessage("2024-1234 어디쯤")` + `AssistantMessage("…역삼역…8시 49분 36초…")`가 그대로 찍힌다(실측 메시지 순서는 `[주입 User, 주입 Assistant, System, 현재 User]` — `MessageChatMemoryAdvisor`가 이력을 배열 *앞*에 끼운다). order만 바꿔 `User 1→2 · Assistant 0→1`이 된 것이 곧 *주입이 로그에 노출된* 증거다.

> **교훈**: 관측 지점의 위치가 곧 관측 대상이다. 메모리 주입을 *로그로* 보려면 로깅 advisor를 memory advisor보다 **뒤(높은 order)** 에 둬야 한다. (단, 메모리 *동작 자체*의 증거는 order와 무관하게 `/messages` 저장·응답의 맥락 해소·세션 격리·삭제 소실이다.)

**전체 advisor 체인 order** (낮을수록 바깥=먼저): `memoryAdvisor(10)` → `SimpleLoggerAdvisor(20)` → `PerformanceLoggingAdvisor(100)`. 따라서 **로깅·성능 측정 모두 메모리 주입 *이후*에 실행**된다 — `PerformanceLoggingAdvisor`(order 100, 가장 안쪽)는 주입이 끝난 프롬프트로 LLM을 호출하고 그 응답의 usage로 토큰을 센다. 실측에서 턴2 `입력 토큰 2621`은 위 SimpleLogger가 찍은 *주입된* 프롬프트(System + 주입 User/Assistant + 현재 User)에 대한 카운트다.

> **토큰 수 해석 주의**: 같은 로그에서 턴1 `입력 토큰 5137` > 턴2 `2621`인데, 이는 메모리 때문이 아니라 **턴1이 `getDeliveryStatus` Tool 왕복**을 거쳐 tool 정의·호출·결과가 프롬프트에 들어갔기 때문이다. *메모리가 더하는 토큰*을 보려면 Tool을 안 부른 턴끼리 비교해야 한다(2단계의 A 실험 2516→3262 선형 증가가 그 깨끗한 측정).

### 설계 결정 문서

#### 1. `MAX_MESSAGES = 20` 의 근거 (계산)

- **단위**: 1턴 = USER 1 + ASSISTANT 1 = **2 메시지**. 따라서 `20 메시지 ≈ 10턴`(USER 10 + ASSISTANT 10).
- **도메인 가정**: 배달 상담은 보통 **3~6턴**(주문 확인 → 상태 조회 → 조치 요청 → 마무리)에 끝난다. 6~12 메시지.
- **계산**: 평균 상담(6~12 메시지)을 **여유 있게 포괄**(20 > 12). 지시 대명사("그거")는 *직전 1~2턴*만 있으면 해소되므로(시나리오 1·3에서 확인), 20은 *필요량 대비 충분 + 안전 마진*.
- **상한을 둔 이유**: 무제한이면 매 요청에 전체 이력이 입력 토큰으로 누적 전송 → 비용·지연 증가(Round 2에서 입력 토큰 4,000+ 관측). Tool 호출 턴은 `TOOL` 메시지도 윈도를 차지하므로 *유효 대화 턴*은 10턴보다 약간 줄어든다 → 20은 그 손실까지 흡수하는 마진.
- 초과분은 `MessageWindowChatMemory`가 **최근 20개만 유지**(오래된 것부터 절삭).

#### 2. `X-Session-Id` 미존재 시 `"default"` 폴백의 위험 시나리오

| # | 시나리오 | 결과 |
|---|---|---|
| (a) | **구버전 클라이언트** — 헤더 미구현 앱이 업데이트 안 됨 | 그 앱의 *모든 고객*이 단일 `default` 세션을 공유 → 서로 다른 사람의 대화가 한 conversationId에 누적 → **타인 주문번호·주소가 다음 사람 응답에 노출**(PII 유출)·맥락 오염 |
| (b) | **어뷰저가 의도적으로 헤더 제거** | 헤더를 빼 `default` 풀에 합류 → *다른 고객의 직전 대화*(주문번호·주소 언급)를 엿봄 → 정보 유출. 역으로 `default`를 의도적으로 오염시켜 타인 응답 교란 |
| (c) | **프록시/LB의 헤더 누락** | 게이트웨이가 커스텀 헤더를 스트립하면 무관한 사용자들이 일제히 `default`로 합류 → (a)와 동일 사고가 *대규모로* 발생 |

→ `default` 폴백은 **데모·로컬 편의용**일 뿐, 프로덕션에선 *폴백 자체를 금지*(헤더 없으면 401/익명 임시세션 발급)해야 한다.

#### 3. 세션 식별 실무 대안 4가지 비교 (배달 상담 도메인 기준)

| 방식 | 동작 | 일반 장점 | 일반 단점 |
|---|---|---|---|
| **쿠키** (`Set-Cookie: sid`) | 서버 발급, 브라우저 자동 전송 | `HttpOnly`+`Secure`로 XSS 탈취 완화, 서버가 값을 통제 | 모바일 앱·서버간 호출엔 부자연, CSRF 방어 필요 |
| **JWT 클레임** (`sub`) | 인증 토큰의 서명된 클레임에서 세션키 도출 | **서버 서명 → 위조 불가**, 인증과 세션키 일원화 | 토큰 만료·갱신 관리, conversationId를 클레임에 고정하면 회전 어려움 |
| **URL 경로** (`/session/{id}`) | 경로 파라미터로 명시 | 디버깅·운영 API엔 명시적이라 편함(본 `SessionController`가 이 방식) | URL이 로그·Referer·공유 링크에 남아 **유출**, 추측·열거 쉬움 |
| **HTTP 헤더** (`X-Session-Id`, 현재) | 클라이언트가 헤더로 전달 | 구현 단순, 엔드포인트·응답타입 무관(assistant/support 공용) | **클라이언트가 임의 지정 → 위조·추측 위험**(아래 4번) |

> 위 장단점은 *일반론*이다. 같은 방식도 **배달 상담 도메인**에 놓으면 가중치가 달라진다. 도메인 특성 — ① 주 채널이 **네이티브 모바일 앱** ② **주문=로그인+결제**라 로그인 고객 비중이 높음 ③ 전화번호 기반 **게스트 주문조회** 병존 ④ 상담은 대개 *특정 주문 1건*에 관한 것 ⑤ **주문번호가 순번(2024-1234…)이라 추측 쉬움** ⑥ 앱→챗봇→상담사 **옴니채널 인계** ⑦ 주소·전화·결제 **PII 다수** — 에 비춘 재평가:

| 방식 | 배달 상담 도메인에서의 가중치 |
|---|---|
| **쿠키** | ▼ 약화 — 주 채널이 **네이티브 앱**이라 쿠키 자동전송 모델이 안 맞는다. 웹 "주문조회"(게스트 추적) 페이지에만 부분 적합. |
| **JWT `sub`** | ▲ 강화 — 로그인 고객 비중이 높아 `sub`가 자연스러운 세션 키. 앱→웹→상담사로 채널이 바뀌어도 같은 `sub`로 **상담 연속성**이 유지되고, "이 주문이 이 고객 것인가"라는 **주문 소유권 검증**과 한 줄로 묶인다. |
| **URL 경로** | ▼▼ 특히 위험 — 주문 상세엔 주소·전화가 붙으므로 세션ID가 **카톡 공유링크·푸시 딥링크·상담 화면 캡처**에 남으면 PII 유출로 직결. 단 푸시알림이 *특정 주문 1건* 페이지로 보낼 땐 명시성이 장점이 되는 양면성. |
| **HTTP 헤더** | ▼ 위험 증폭 — **주문번호가 순번이라 추측이 쉽다**. 클라이언트가 세션키를 자유 지정하면 타인 주문 대화 하이재킹 난이도가 낮다(본 실험의 `s5r1`류 세션명이 추측 가능했던 게 곧 증거 — 아래 4번). |

→ **배달 도메인 최적안 — `sub`(고객) + `orderId`(주문) 결속**:
상담이 거의 항상 *특정 주문*에 관한 것이라는 점을 활용해 `conversationId = hash(sub + orderId)`로 두면, (1) 한 주문의 **전체 상담 라이프사이클**(상태조회→취소→환불→재배달)이 한 대화로 묶이고, (2) **동시 진행 중인 여러 주문**을 주문별로 분리하며, (3) `sub`가 그 `orderId`의 소유자인지 서버가 검증해 **IDOR 자동 차단**(4번), (4) 지시 대명사 "그거"가 *그 주문 맥락 안*에서만 풀려 모호성이 준다.
다만 고객이 한 창에서 여러 주문을 섞어 물으면("1234랑 1235 둘 다") 주문 경계가 깨지므로 — 현실적으로는 **세션키는 `sub` 단위로 두고, 주문 식별(orderId)은 Tool 파라미터로 해소**하는 하이브리드가 더 안전하다(메모리는 *고객 단위*, 주문 식별은 *Tool 책임*).

→ **권장 정리**: 로그인 고객은 **JWT `sub` 기반**(+ `orderId` 소유권 검증), 게스트 주문조회는 **HttpOnly 쿠키 또는 서버 발급 1회성 토큰**. 현재의 `X-Session-Id`는 *학습·관찰용*이며, 특히 **주문번호 추측 가능성** 때문에 그대로 운영에 쓰면 안 된다.

#### 4. 세션 ID를 **클라이언트가 직접 정하게** 하는 방식의 보안 리스크와 방어

- **리스크 (IDOR/세션 하이재킹)**: 현재 `X-Session-Id`는 클라이언트가 값을 자유 지정한다. 공격자가 `cust-A`, `s5r1`, 순번 등 **추측·열거**로 타인 conversationId를 지정하면 그 사람의 대화 이력을 *조회·오염*할 수 있다(본 실험의 세션명이 추측 가능했다는 점이 곧 취약점).
- **방어**:
  1. **서버 발급** — 클라이언트가 못 정하게 하고 서버가 **추측 불가능한 ID**(`UUIDv4` 등)를 발급해 내려준다.
  2. **인증 바인딩(소유권 검증)** — conversationId를 인증 주체(JWT `sub`)에서 *도출*하거나, 요청의 세션키가 *인증된 사용자 소유*인지 서버가 매 요청 검증한다.
  3. **JWT 서명 검증** — 세션키를 **서명된 토큰**에 담아 위조를 차단(클라이언트가 바꾸면 서명 깨짐).
- **핵심 원칙**: *세션키 = 서버가 통제하는 인증 파생값*. 클라이언트의 자유 입력을 신뢰하지 않는다.

#### 인프라 결정 (참고)

- **InMemory + `@Profile("!jdbc")`** — 개발 기본 InMemory(재시작 시 소멸 허용). 영속화·멀티 인스턴스·감사가 필요하면 jdbc 프로파일에서 자동 구성된 `JdbcChatMemoryRepository`로 교체(코드 변경 없음). 전환 기준·스키마·PII 주의는 `JdbcChatMemoryExample` 참조.

## 2단계: Memory 크기 실험 + 실패 관찰

**목표**: `MAX_MESSAGES`를 바꿔 가며 같은 10턴 시퀀스를 돌리고, 입력 토큰 / 정확도를 정량 비교한다.

### 구현

`MAX_MESSAGES`를 컴파일 상수에서 **런타임 프로퍼티**로 빼 재컴파일 없이 값을 바꾼다(기본값 20 유지):

```java
@Bean
ChatMemory chatMemory(ChatMemoryRepository repo,
                      @Value("${chat.memory.max-messages:20}") int maxMessages) {
    return MessageWindowChatMemory.builder()
            .chatMemoryRepository(repo).maxMessages(maxMessages).build();
}
```
```bash
./gradlew bootRun --args='--spring.profiles.active=local --chat.memory.max-messages=2'        # 실험 B
./gradlew bootRun --args='--spring.profiles.active=local --chat.memory.max-messages=2147483647' # 실험 C
```

동일 10턴 시퀀스(`expA`/`expB`/`expC` 세션): `1) 1234 배달상황 → 2) "그거 몇 분?" → 3) 1235 메뉴 → 4) "아 그 버거세트" → 5) 1234 취소가능? → 6) "그럼 1235는?" → 7) "그거 취소"(=1235) → 8) "아까 1234는?" → 9) "그 주문 라이더"(=1234) → 10) "지금까지 요약"`.

### 정량 비교

#### (1) A·B·C 종합 — 동일 10턴 지시-대명사 시퀀스 (caffeinate 재실측)

| 실험 | MAX_MESSAGES | 10턴 평균 입력 토큰 | 10턴 평균 출력 토큰 | 10턴 평균 응답 시간† | 지시·참조 해결‡ |
|---|---|---:|---:|---:|---:|
| **A** | 20 | 3,262 | 97 | ~14.1s | **7/7** |
| **B** | 2 | 2,597 | 56 | ~7.8s | **2/7** |
| **C** | MAX_VALUE | 3,202 | 87 | ~10.8s | **7/7** |

> † 응답시간은 로컬 wall-clock(T1 모델 콜드로드·Tool 왕복 포함)이라 *참고용* — 결정론 지표는 입력 토큰.
> ‡ 지시·참조 턴은 10턴 중 7개(T2·T4·T6·T7·T8·T9·T10). T1·T3·T5는 새 주문번호를 직접 제시(참조 아님). C의 T6·T7 오답은 1235를 "존재하지 않음"으로 답한 *Round2 tool 환각*이며 참조 해결 자체는 정확. B의 실패 5턴은 모두 "주문번호를 알려달라"/"찾을 수 없음".

**읽는 법**: **A와 C는 입력토큰·해결률이 사실상 동일**(10턴=20메시지라 C도 *아직 cap 미도달*) — 둘의 차이는 *11턴 이후*에야 드러난다(아래 (2)). 반면 **B(2)는 토큰만 최소(2,597)이고 참조 해결이 2/7로 붕괴**(직전 1턴 참조 T2·T4만 성공). 즉 *너무 작으면 맥락이 깨지고, 충분하면(20·∞) 10턴 구간에선 차이가 없다*.

#### (2) A vs C — 11턴 이후 발산 (14턴 재실측, MAX=20 cap 검증)

> (1)에서 A=C인 건 윈도가 *아직 cap에 안 닿았기* 때문. cap이 실제로 갈리는 지점을 보려고 14턴까지 늘려 측정(Tool 미유발 정책문의, 입력토큰=결정론 지표).

| 턴(누적 메시지) | A=20 입력토큰 | A 윈도 | C=∞ 입력토큰 | C 윈도 |
|---|---:|---:|---:|---:|
| T1 (2) | 2,514 | 2 | 2,514 | 2 |
| T5 (10) | 2,895 | 10 | 2,868 | 10 |
| T10 (20) | 3,471 | **20** | 3,406 | 20 |
| T11 (22) | 3,575 | **20** | 3,479 | **22** |
| T12 (24) | 3,620 | **20** | 3,612 | **24** |
| T13 (26) | 3,704 | **20** | 3,727 | **26** |
| T14 (28) | 3,721 | **20** | 3,842 | **28** |

**C 입력토큰은 cap 없이 선형 증가** — 14턴 전체:
`2,514 → 2,622 → 2,703 → 2,808 → 2,868 → 2,965 → 3,070 → 3,161 → 3,266 → 3,406 → 3,479 → 3,612 → 3,727 → 3,842`(28메시지). A는 20메시지(T10)에서 멈춰 이후 평탄(3,471→3,721, 윈도 churn으로 출렁일 뿐). (전체 수치 raw 참조.)

> 한 줄 요약: **MAX_MESSAGES는 "윈도(메시지 개수)" 하나의 레버 → 세 축(비용·맥락충실도·지연)을 trade-off, 그리고 언어 드리프트와는 직교**. 아래 4 Finding으로 분해한다.

### Finding 1 — 윈도는 "메시지 *개수*"를 자른다 (토큰·의미가 아니라)

- **≤10턴(≤20메시지)**: A와 C가 거의 동일(2,514 → ~3,400). 윈도(20)가 *아직 cap에 안 닿아* 둘 다 전부 보관 → 같은 곡선(재실측 T1~T10 일치).
- **11턴+**: 분기. **A는 윈도가 20개에서 고정**(T11~T14 모두 win=20), **C는 22→24→26→28로 무한 증가**. 토큰도 A는 +250(둔화), C는 +436(단조 상승).
- **단, A도 완전 평탄은 아니다**: 윈도는 *메시지 20개*를 자르지 *토큰*을 자르지 않는다 → 보관된 20개 메시지의 길이에 따라 토큰이 출렁인다(A의 T11~T14 증가폭이 +104·+45·+84·+17로 들쭉날쭉한 이유). **"메시지 cap ≠ 토큰 cap"** 이 핵심.

> MAX=20을 고른 근거: 전형적 10턴 상담이 *cap에 막 안 닿는* 지점 — 정상 세션에선 20과 ∞가 비용상 동일하고, ∞는 이득 없이 장기 폭증 리스크만 더한다(Finding 3).

### Finding 2 — 작을수록(B=2) 맥락이 *구조적으로* 붕괴한다

`MAX_MESSAGES=2` = USER 1 + ASSISTANT 1 = **직전 한 번의 교환만** 윈도에 남는다.
- *직전 턴* 지시 대명사는 성공(T2→T1, T7→T6 — 그 쌍이 윈도에 있음).
- 하지만 **두 다리 이상 참조·전체 이력 집계는 물리적으로 불가능**.

**캡처 — T10 요약 붕괴**: `"요약해 주세요 지금까지 제가 뭘 물어봤는지"` →
*"…'그 주문 라이더 위치 다시 확인'이라는 내용으로…"* — **T1~T8 전부 누락, 직전 T9만 요약**. 윈도 = `[T9 USER, T9 ASSISTANT]` 뿐이라 T1~T8이 *프롬프트에 존재하지 않는다*.
```json
[{"type":"USER","content":"요약해 주세요 지금까지 제가 뭘 물어봤는지"},
 {"type":"ASSISTANT","content":"... 앞서 말씀하신 내용을 정리하면:\n- \"그 주문 라이der 위치 다시 확인\"이라는 내용으로 ..."}]
```
> 같은 요청을 A(20)는 5개 항목 전부 정확 요약. **동일 코드·동일 발화인데 윈도 크기만으로 의미 불능**이 된다.

### Finding 3 — 클수록(C=∞) 비용·지연이 단조 증가한다

- **토큰/비용**: C는 매 턴 전체 이력을 입력으로 재전송 → 입력토큰이 cap 없이 단조 증가(28메시지에서도 계속 상승). A는 20메시지에서 평탄 → **장기 세션일수록 C의 비용이 무한히 벌어진다.**
- **지연 — 왜 둔화되나(메커니즘)**: LLM 1콜 = **prefill**(입력 N토큰 전부 인코딩·KV캐시 구축, 비용 ∝ N) + **decode**(출력 토큰 생성). C는 입력 N이 매 턴 커지므로 **prefill 시간이 단조 증가**, A는 평탄. 출력 토큰은 턴마다 비슷하니 지연 증가분은 prefill(=입력 토큰)이 지배한다.
- **지연 — 로컬 측정은 노이즈가 크다**(A·C 모두 7~75s 출렁, T1~T14는 안 갈림). 30턴까지 늘리면 **T15부터 C가 67~114s로 폭증**해 A(15~50s)와 확연히 갈라지지만(→ *실패 관찰 (3)*), 그 폭은 입력토큰 증가분만으론 과소예측된다 — *prefill ∝ N* + 긴 컨텍스트의 머신/추론 비선형이 겹친 것으로 보인다. 결정론 지표는 여전히 입력 토큰.

### Finding 4 — 언어 드리프트(중국어 폭주)는 메모리 크기와 **직교**한다

윈도 크기와 무관하게 발현하는 *모델 레벨 확률적 결함*. 재실측이 통제 조건으로 확정:

| 윈도 | 드리프트 | 근거 |
|---|---|---|
| A2 = 20 | **발생** (T4·T9·T14) | 한국어→중국어 폭주 |
| C2 = ∞ | **0건** | 동일 14프롬프트인데 깨끗 |
| B = 2 | 발생 (T9) | 중국어 폭주 |
| 시나리오5 = 20 | 발생 (R1) | 중국어+일본어 혼입 |

→ 윈도 2·20·∞ 전부에서 관측되고, **같은 모델·같은 프롬프트인데 A2는 폭주·C2는 정상**(temp=0.3) = 윈도와 상관 없는 **확률적** 실패.

**왜 중국어인가 (결과론 아닌 메커니즘)**:
1. **모델 출신** — qwen2.5(Alibaba)는 중국어가 사전학습 지배 언어 → 토큰 확률질량 최상위, 한국어는 상대적 저자원.
2. **진입점 = 한자어(사이노코리안) 경계** — 한국어 어휘 다수가 한자어라 `주문이력→历史记录`·`고객센터 전화→拨打客服电话`·`결제수단→支付方式`처럼 *동의 개념의 중국어 토큰 빈도가 압도적*이다. 그 지점에서 한국어 토큰을 이긴다. (재실측 드리프트가 정확히 이 단어들에서 시작 — raw 참조)
3. **자기회귀 cascade** — 중국어 토큰 1개가 나오면 "직전과 같은 언어" prior가 급등 → 문장 전체 runaway = "폭주". 출력이 길수록 심함(A2 T14 요약이 가장 전면적).
4. **지시 prior 약화** — "항상 한국어" 시스템 룰은 *소프트 prior*라 토큰 prior와 경쟁하다 진다(명시 룰에도 폭주).

**실제 해결책**(프롬프트 아님): 제약 디코딩/언어 고정 샘플링, 한국어 강세 모델(EXAONE·HyperCLOVA 등), 출력 언어 검증 후 재생성 가드.

**검증 가능한 예측**: 드리프트는 한자어/외래어 토큰에 집중(✓), MAX_MESSAGES와 무관(✓ 2·20·∞), 출력 길이에 비례(✓ T14), 한국어 강세 모델로 교체 시 소멸(미검증·다음 단계).

### 실패 관찰 — MAX=2 파괴 · C 지연 (출력 그대로 기록)

> 형식적 "안 됐어요"가 아니라 *시스템이 어떻게 망가지는지* 출력 자체를 남긴다. (위 Finding의 raw 캡처 층.)

**(1) B(MAX=2) — 지시 대명사가 해소되지 못한 턴 캡처**
- 요청(T9): `"그 주문 라이더 위치 다시 확인"`
- LLM 응답: *"라이더의 현재 위치를 확인하겠습니다. **주문번호를 알려주실 수 있으신가요?** 예를 들어 \"1234\"와 같은 형식으로…"* — **"그 주문"(=1234)을 못 끌어와 되묻는다.**
- T9 시점 Memory(`/messages`) — 윈도엔 **직전 1쌍(2건)뿐**(1234 맥락은 이미 절삭):
```json
[{"type":"USER","content":"아까 1234는 언제 도착해요?"},
 {"type":"ASSISTANT","content":"배달 상태를 확인해 보겠습니다.\n\n주문번호를 알려주실 수 있으신가요? 예를 들어 \"1234\"와 같은 형식으로 주문 번호를 말씀해주시면 ..."}]
```
- 같은 붕괴 T7: `"그거 취소해주세요"` → *"취소하려는 주문번호를 알려주실 수 있으신가요?"* ("그거"=1235 소실). (원본 B T9는 아예 중국어로 폭주 → Finding 4.)

**(2) 왜 2는 부족한가 — USER/ASSISTANT 메시지 구조 기준**
- `MAX_MESSAGES=2` = **USER 1 + ASSISTANT 1 = 직전 한 번의 교환**만 남는다.
- T9 윈도 = `[T8 USER, T8 ASSISTANT]` 뿐 → "그 주문"이 가리키는 1234(T1에서 등장)가 *프롬프트에 물리적으로 부재*. 모델이 아무리 잘해도 없는 정보는 못 끌어온다.
- 직전 1턴 참조(T2→T1, T4→T3)는 성공하지만 *두 다리 이상*·*전체 요약*은 구조적 불가(T10 요약이 직전 1건만 — Finding 2).

**(3) C(MAX=∞) — 응답 시간이 A 대비 언제부터 느려지나 (A·C 30턴, LLM 처리시간 ms)**

로컬 처리시간은 노이즈가 크다 — **T1~T14는 A·C가 안 갈린다**(둘 다 콜드로드·랜덤 스파이크로 7~75s 출렁; T11~15 블록은 오히려 A가 더 느림 — A의 T12=74.6s 스파이크). 그러나 **T15부터 C가 폭증**한다. 5턴 블록 평균 처리시간(실측):

| 턴 블록 | A=20 | C=∞ |
|---|---:|---:|
| T1–5 | 24.5s | 26.3s |
| T6–10 | 9.8s | 13.0s |
| T11–15 | 37.1s | 28.1s |
| T16–20 | 18.7s | **84.7s** |
| T21–25 (24까지) | 26.9s | **103.3s** |

- **분기 턴 = 약 T15**: C가 67→77→88→…→114s로 치솟고, A는 15~50s 밴드 유지. C는 24턴까지 단조 상승(턴당 100s+로 느려져 30턴 전 측정 중단).
- **단, 토큰만으론 과소예측**: C 입력토큰은 ~4,090(T24)인데 처리시간 100s+ — A(~3,800토큰/~30s)보다 토큰은 조금 큰데 시간은 3~4배. *prefill 증가(Finding 3)에 더해, 긴 컨텍스트에서 머신/추론 처리의 비선형 악화*가 겹친 것으로 보인다. **깨끗한 prefill∝N 곡선은 아니다.**
- (전체 30턴 턴별 토큰·ms는 raw 참조. 초반 노이즈가 커 "정확히 몇 번째 턴"은 ±1~2턴 폭으로 본다.)

**→ 언제·왜·그래서 무엇을 아는가**
- **언제**: A 대비 ~**T15**(컨텍스트가 충분히 커진 뒤)부터 C가 갈라져 67→114s로 폭증.
- **왜**: 입력이 cap 없이 누적 → 매 턴 prefill(입력 전체 인코딩)이 커짐(이론상 ∝N). 그런데 실측은 그보다 *더 가파른 비선형* — 로컬 단일 머신에서 컨텍스트가 커질수록 KV캐시·메모리 대역폭·스레드 경합이 겹쳐 선형 이상으로 악화.
- **그래서 알 수 있는 것**:
  - 무제한 메모리는 *비용(토큰)은 선형으로 예측*되지만 **지연은 어느 임계부터 비선형으로 폭발**한다 → 장기 세션에서 *체감 응답성이 갑자기 무너지는* SLA 리스크.
  - 즉 윈도 cap(A=20)은 비용 절감뿐 아니라 **지연 안정성(predictable latency)** 장치다 — cap이 있으면 프롬프트가 평탄해 지연이 노이즈 밴드 안에 머문다.
  - 로컬 qwen2.5라 임계가 일찍(~15턴) 왔을 뿐, 프로덕션 GPU·서버면 임계 턴은 뒤로 밀려도 *원리는 동일* → **무제한 금지, cap 또는 요약 하이브리드(설계 결정 1) 필수.**

### 설계 결정 문서

#### 1. 슬라이딩 윈도 vs 요약(summarization) 전략

| 전략 | 동작 | 장점 | 단점 | 적합 시나리오 |
|---|---|---|---|---|
| **슬라이딩 윈도**(현재 `MessageWindowChatMemory`) | 최근 N개 메시지만 유지 | 단순·**결정론적**·저지연, 추가 LLM 호출 없음 | 윈도 밖 정보 **완전 소실**(장기 맥락·전체 요약 불가) | 짧은 단발 상담(배달 상태·취소처럼 1세션에 끝나는 흐름) |
| **요약** | 오래된 메시지를 LLM이 압축 요약본으로 치환해 유지 | 장기 맥락 보존 + 토큰 절약, "지금까지 요약" 가능 | 요약용 LLM 호출 **추가 비용·지연**, 요약 **손실·왜곡**(비결정적), 구현 복잡 | 긴 멀티턴/재방문 상담(누적 클레임, 반복 고객 이력) |

→ **배달 상담은 대부분 단발**(상태 조회·취소)이라 슬라이딩으로 충분. 단, *전체 요약*이나 *장기 클레임 추적*이 필요하면 **하이브리드**(최근 N개 원문 + 그 이전은 요약본)가 정답. MAX=2 실험이 보여준 "요약 붕괴"는 *슬라이딩만으로 장기 요약을 기대하면 안 된다*는 증거.

#### 2. 프로덕션 금지 기준 — 지시 대명사 해결률

- **내 기준**: 핵심 참조 턴에서 **80% 미만이면 프로덕션 금지**. 단, **변경 작업(취소·환불)을 동반한 지시 대명사**는 *단 1건의 오해도 불가* → 사실상 100% 또는 **명시적 재확인 강제**.
- **근거**: 조회 오류("그거 어디쯤?"을 다른 주문으로 답)는 재질문으로 복구 가능하지만, **"그거 취소해줘"를 엉뚱한 주문에 적용하면 금전·신뢰 사고**(잘못된 주문 취소 → 환불 분쟁·정산 오류). 실험에서 B는 3/7(43%)로 **명백한 금지선 아래**, A는 6/7(86%)로 조회는 통과하나 *취소 동반 턴은 재확인 게이트 필수*.

#### 3. 배달 도메인에서 "오래된 대화"가 의미 있는 경우

- **수 시간 전 주문 재문의**: *"아까 그 치킨 아직도 안 왔어요"* (3시간 전 주문 건).
- **어제 처리 건 재문의**: *"어제 환불 신청한 거 어떻게 됐어요?"* — 윈도 밖이라 세션 메모리로는 불가, 영속 이력 필요.
- **반복 고객 선호**: *"늘 시키던 걸로"* — 과거 주문 패턴.
- **분쟁/클레임 맥락**: 이전 CS 이력이 있어야 일관된 응대 가능.
→ 이들은 *세션 윈도*가 아니라 **고객 단위 영속 이력**의 영역.

#### 4. Memory를 세션이 아닌 **고객 단위로 영속** 유지하면

- **가능해지는 기능**: 주문 이력 기반 개인화(*"지난번 그 메뉴 또 드릴까요?"*), 재방문 시 직전 클레임 맥락 자동 로드, 선호·알러지 학습, 세션 끊겨도 대화 연속.
- **리스크**:
  - **PII 장기 축적** — 주소·전화·결제 정보가 무기한 평문 누적 → 유출 시 피해 규모 폭증(5주차 Guardrail의 입력 마스킹 필요).
  - **법적 의무** — 개인정보보호법/GDPR의 *삭제권·보존기간 제한*. 영속 메모리엔 TTL·파기 정책 필수.
  - **저장 비용 무한 증가** — 고객×대화 누적 → 파티셔닝·아카이브·요약 압축 필요(`JdbcChatMemoryExample`의 운영 고려사항).
  - **stale 오답** — 오래된 정보로 잘못 안내(*"작년 주소로 배달 안내"*).
  - **보안 표면 확대** — 영속 저장소가 새 공격 대상.

## 3단계: InMemory vs JdbcChatMemory — 의사결정 트리

**목표**: JDBC 저장소로 전환하고, 어떤 운영 조건에서 어느 저장소가 정답인지 의사결정 트리를 그린다.

### 구현

- `build.gradle`: `spring-ai-starter-model-chat-memory-repository-jdbc` + `runtimeOnly com.h2database:h2`.
- `ChatMemoryConfig.chatMemoryRepository()` 에 `@Profile("!jdbc")` — jdbc 프로필에선 이 InMemory 빈을 안 만들고, 자동 구성된 `JdbcChatMemoryRepository`가 주입된다.
- 실행: `./gradlew bootRun --args='--spring.profiles.active=jdbc'`

> **셋업 이슈 2건 — 직접 해결(그 자체가 finding)**
> 1. **Spring AI 1.0.0은 H2 스키마를 패키징하지 않는다** — jar 안엔 `schema-postgresql/sqlserver/hsqldb/mariadb.sql`만 있고 `schema-h2.sql`이 없다. H2 datasource는 platform=`h2`로 감지돼 `classpath:.../schema-h2.sql`을 찾다 *No schema scripts found*로 기동 실패. → **`src/main/resources/org/springframework/ai/chat/memory/repository/jdbc/schema-h2.sql`을 직접 추가**(postgresql 스키마와 동등, H2 호환).
> 2. **`initialize-schema: embedded`는 *in-memory* DB만 초기화한다** — `application-jdbc.yml`의 기본 `embedded`는 `h2:mem`에선 테이블을 만들지만, **`h2:file`(영속)에선 만들지 않아** `Table "SPRING_AI_CHAT_MEMORY" not found` 500. → file 모드는 **`--spring.ai.chat.memory.repository.jdbc.initialize-schema=always`** 필요.

### JDBC 프로필에서 1단계 시나리오 5종 재실행 — 저장소 투명 교체

`--spring.profiles.active=jdbc`(h2:mem)에서 1단계 5종을 그대로 재실행. **저장소만 `InMemoryChatMemoryRepository` → 자동구성 `JdbcChatMemoryRepository`로 바뀌고, 상위 `MessageWindowChatMemory`/`MessageChatMemoryAdvisor`는 그대로** → 메모리 동작이 1단계와 같아야 한다. (qwen2.5는 비결정적이라 응답 *문장*은 매번 다르므로, **메모리가 책임지는 동작**의 동일성으로 판정.)

| # | 시나리오 | JDBC(h2:mem) 실측 결과 | 메모리 동작 |
|---|---|---|:--:|
| 1 | "그거 언제 도착?" | T2 *"배달 예정 시간을 확인해드리겠습니다. 주문번호 **2024-1234**의 배달 상태를…"* — 이전 턴 1234 맥락 유지(이번엔 Tool 미호출) | ✅ |
| 2 | "그거 말고 1235 취소" | T2 *"주문 번호 **2024-1235**의 취소 요청을 처리하겠습니다."* — 대상 전환 | ✅ |
| 3 | "아까 물어본 그 주문" | T2가 `{"name":"getDeliveryStatus","arguments":{"orderId":"**2024-1234**"}}` 로 호출 — 1234 참조 성공. *단 raw tool-call JSON이 본문에 누출*(`ONGL …</tool_call>`) | ✅(참조)·포맷깨짐 |
| 4 | 세션 격리(A→B) | B *"주문번호를 알려주시면…예: 2024-1234"* — **A의 1234 전혀 모름**. (A 응답엔 `라이ダー` 드리프트) | ✅ |
| 5 | 삭제 후 "그거" | `DELETE`→HTTP 200→`/messages`=`[]`→T2 *"주문번호를 알려주시면…"* **맥락 소실** | ✅ |

- `/session/ids` = `["js1","js2","js3","js4A","js4B","js5"]` — 6개 세션 전부 **JDBC 저장소**에 독립 등록(세션 격리도 저장소 무관).
- 시나리오 1 `/messages`(JDBC에서 4건 read, **실측 그대로**):
```json
[{"type":"USER","content":"2024-1234 어디쯤 있어요?"},
 {"type":"ASSISTANT","content":"배달 상태를 확인해드리겠습니다.\n\n주문번호 2024-1234의 배달 진행 상황을 알려주시겠어요?"},
 {"type":"USER","content":"그거 언제 도착해요?"},
 {"type":"ASSISTANT","content":"배달 예정 시간을 확인해드리겠습니다.\n\n주문번호 2024-1234의 배달 상태를 알려주시겠어요?"}]
```
- 시나리오 5 — 삭제 전 `[USER, ASSISTANT]`(1쌍) → `DELETE js5`(HTTP 200) → 삭제 후 `[]` → "그거" 재질문에 주문번호 되물음.

> **결론**: 5종 전부 메모리 동작이 1단계와 **동일**(맥락 유지·대상 전환·orderId 참조·세션 격리·삭제 소실) — *저장소 계층만 교체*했음을 확인. 동시에 **tool-call JSON 누출(S3)·언어 드리프트(S4 `라이ダー`)도 1단계처럼 재현** → 이 둘은 *저장소와 무관한 모델 레이어 변동*(Round 2·Finding 4와 일치). 즉 JDBC 전환은 메모리 동작을 그대로 보존하고, 모델 변동성도 그대로 둔다.

### H2 SQL — `SELECT * FROM SPRING_AI_CHAT_MEMORY ORDER BY "timestamp"` (5종 데이터, h2:file)

위 **5종을 `h2:file`(`;WRITE_DELAY=0`)로 적재** → 앱 종료 후 H2 Shell로 파일 DB 직접 조회. 5종 6세션 = **20행**이 그대로 영속(세션별: js1 6 · js2 4 · js3 4 · js4A 2 · js4B 2 · js5 2). 아래는 H2 Shell 실측 출력(content는 34자 head):

```text
CID  | TYPE      | CONTENT(head)
js1  | USER      | 2024-1234 어디쯤 있어요?
js1  | ASSISTANT | 배달 상태를 확인해드리겠습니다. … 주문번호 2024-1234의
js1  | USER      | 그거 언제 도착해요?
js1  | ASSISTANT | 배달 예정 시간을 확인해드리겠습니다. … 주문번호 2024-12…   ← "그거"에 1234 유지
js1  | USER      | 아까 그 주문 어디쯤이었죠?                              ← 재시작 후 턴
js1  | ASSISTANT | 주문번호 2024-1234의 배달 위치와 상태를 확인해드리겠습…   ← 복원된 1234
js2  | USER      | 2024-1234 취소해주세요
js2  | ASSISTANT | 주문이 취소됩니다. 확인 후 다시 안내드리겠습니다. 주문 번…
js2  | USER      | 아, 그거 말고 2024-1235 취소해주세요                     ← 대상 전환
js2  | ASSISTANT | 주문이 취소됩니다. 확인 후 다시 안내드리겠습니다. 주문 번…
js3  | USER      | 2024-1234 어디쯤 있어요?
js3  | ASSISTANT | 주문이 배달 시작 단계에 있습니다. 현재 라이더는 역삼…
js3  | USER      | 아까 물어본 그 주문 언제 도착해요?
js3  | ASSISTANT | 주문이 배달 시작 단계에 있습니다. 현재 라이더는 역삼…     ← 1234 배달맥락 이음
js4A | USER      | 2024-1234 어디쯤 있어요?
js4A | ASSISTANT | 배달 상태를 확인해드리겠습니다. … 주문번호 2024-1234의
js4B | USER      | 그 주문 어디쯤이에요?
js4B | ASSISTANT | 배달 진행 상황을 확인해드리겠습니다. 주문번호를 알려주…   ← 격리(1234 모름)
js5  | USER      | 그거 언제 도착해요?
js5  | ASSISTANT | 배달 상태를 확인해 보겠습니다. 주문번호를 알려주시면 정확…  ← 삭제 후 소실

```
- 컬럼: `conversation_id`(세션키)·`content`·`type`(USER/ASSISTANT/TOOL/SYSTEM)·`"timestamp"`.
- **SELECT가 5종 시나리오의 메모리를 DB 레벨에서 그대로 보여준다** — js1(맥락 유지·재시작 복원)·js2(1234→1235 전환)·js3(orderId 이음)엔 주문 맥락이, **js4B(격리)·js5(삭제 후)엔 주문번호가 없다**(되물음). 메모리 동작이 저장소에까지 반영됨.
- (구현 노트) `MessageWindowChatMemory`는 턴마다 대화 전체를 *delete+재삽입*해 윈도를 갱신 → 같은 세션 행들의 `timestamp`가 마지막 저장 시점으로 수렴.

### 재시작 실험 (영속성 검증)

5종 적재(h2:file) → 앱 종료 → 재기동 → 같은 세션 질의. **실측 결과**:

| 저장소 설정 | 재시작 후 유지 | 실측 |
|---|:--:|---|
| **InMemory** (기본 `!jdbc`) | ❌ | JVM 힙 — 프로세스 종료 시 소멸 |
| **`jdbc:h2:mem`** | ❌ | `DB_CLOSE_DELAY=-1`은 *JVM 생존 동안만*. 재시작 후 `/ids`=`[]` |
| **`jdbc:h2:file`** | ✅ | 디스크 파일 영속. 재시작 후 `/ids`=**6세션 복원**·js1 4건 복원, T3 *"아까 그 주문 어디쯤이었죠?"* → *"주문번호 **2024-1234**의 배달 위치와 상태를 확인해드리겠습니다"*(복원된 맥락) |

> ⚠️ **크래시 내구성 주의**: h2:file은 *설계상 영속*이지만 H2 MVStore가 commit을 **비동기 flush**(기본 `WRITE_DELAY`≈500ms)한다. 정상 종료(graceful shutdown)면 셧다운 시 flush되어 복원되지만, **비정상 종료(crash/abrupt kill)** 시엔 flush 전 쓰기가 유실될 수 있다. 본 실험에서도 `pkill` 거친 종료로 적재분이 통째로 날아간 경우가 있었고, **`WRITE_DELAY=0`(commit 즉시 동기 flush)** 로 두니 거친 종료에도 복원됐다. → 크래시까지 견디려면 graceful shutdown 보장 또는 동기 flush 설정.

**관찰 결과**:
- **재시작 후 맥락 연속성 실증.** 앱을 껐다 켠 뒤 T3 *"아까 그 주문"*이 **파일에서 복원된 1234**를 끌어와 응답 — InMemory였다면 불가능. **무중단 배포·오토스케일로 인스턴스가 교체돼도 대화가 끊기지 않는다**가 JDBC file/외부 DB의 핵심 가치.
- **`h2:mem`은 JDBC라도 영속이 아니다 (흔한 오해).** 재시작 후 `/ids`=`[]`. 진짜 영속은 `h2:file` 또는 외부 DB(PostgreSQL 등)부터.
- **영속은 "JDBC면 끝"이 아니다.** 저장소를 JDBC로 바꾼 것과, *크래시에도 데이터가 남느냐*(flush 내구성·백업·복제)는 별개 축 — 운영에선 종료 전략·백업까지 함께 설계해야 한다.
- **메모리 동작은 저장소와 무관하게 그대로다.** 복원된 세션에서도 지시 대명사 해소가 정상 — 상위 `MessageWindowChatMemory`/`Advisor`는 그대로고 저장 위치만 바뀐 것이라는 3단계 전제를 재확인.

### 설계 결정 문서 — 의사결정 트리

| # | 결정 질문 | 배달 상담 서비스의 답 | 근거 (배달 도메인) | 귀결 |
|---|---|:--:|---|---|
| 1 | LB 뒤 **멀티 인스턴스**로 뜨는가? | **YES** | 점심·저녁 피크·프로모션 트래픽 → 스케일아웃 필수. InMemory면 LB가 다른 인스턴스로 라우팅하는 순간 대화가 끊긴다 | JDBC(공유 DB) |
| 2 | **재시작·무중단 배포** 후에도 대화가 이어져야? | **YES** | 잦은 배포·오토스케일 축소로 인스턴스 교체. 상담은 *진행 중 주문(취소·환불·배달지연)*에 관한 것이라 도중 끊기면 고객 재문의·이탈 | JDBC(file/외부 DB) |
| 3 | 상담 이력 **감사·N년 보관** 의무? | **YES**(부분) | 환불 분쟁·정산 오류·민원/갑질 신고 추적에 이력 필요. 단 PII라 보존기간·암호화·파기정책 동반 | JDBC(+보안) |
| 4 | **단일 인스턴스 + 분 단위 짧은 세션**인가? | **NO** | 개별 상담은 분 단위로 짧지만(이 부분만 Yes) **서비스 전체는 단일 인스턴스가 아니다**(1과 모순) | InMemory 탈락 |


**결론**: 1·2·3 = YES, 4 = NO → **배달 상담 서비스는 JDBC(영속 외부 DB) 채택**. InMemory는 *로컬 개발·데모·단일 인스턴스 PoC* 한정.

**InMemory로 충분한 3조건**: ① 단일 인스턴스(스케일아웃 없음) ② 세션이 짧고 재시작 간 유지 불필요(상태 조회·취소처럼 분 단위 완결) ③ 감사·영속 요구 없음(개발/데모/임시 트래픽).

**JDBC가 필요한 3조건**: ① 멀티 인스턴스 — 어느 서버에 붙어도 같은 대화 ② 재시작/무중단 배포 후에도 대화 연속 ③ 상담 이력 감사·분석·N년 보관 의무.

**실제 운영이라면 어떤 DB? → PostgreSQL**:
- **Round 4에서 PgVector를 띄운다** → 같은 PostgreSQL 인스턴스에 *chat memory + 임베딩 벡터*를 공존시켜 인프라를 하나로(운영·백업·접근통제 일원화).
- 관계형 + 풍부한 인덱스/파티셔닝 + `pgvector` 확장. Spring AI가 **postgresql 스키마를 기본 제공**(H2처럼 직접 안 만들어도 됨).
- *Redis/DynamoDB*는 KV라 감사 쿼리·조인이 약하고, *MySQL*은 가능하나 PgVector 시너지에서 PostgreSQL이 우위.

**JDBC 도입 시 함께 볼 비기능 요구사항 3**:
1. **인덱스** — `conversation_id(+timestamp)` 인덱스 필수(세션 조회 성능). Spring AI 스키마에 포함됨.
2. **TTL / 파티셔닝** — 대화는 무한 누적 → 오래된 세션 배치 삭제·아카이브, 시간 기준 파티셔닝으로 테이블 비대화 방지.
3. **암호화(at-rest) + 백업** — 대화에 PII(주소·전화·결제)가 평문으로 남으므로 저장 암호화 + 정기 백업(RPO/RTO) + 접근 통제·감사 로그. (5주차 Guardrail의 입력 마스킹과 병행.)

## 빌드·실행 메모 (공통)

- 시스템 `JAVA_HOME`이 JDK 1.8(무효)로 잡혀 있어 JDK 17 지정 필요:
  `export JAVA_HOME=$(/usr/libexec/java_home -v 17)`
- DEBUG 로그·`SimpleLoggerAdvisor` 활성화: `./gradlew bootRun --args='--spring.profiles.active=local'`
- 메모리 크기 override(2단계): `--chat.memory.max-messages=2` (기본 20)
- JDBC 저장소(3단계): `--spring.profiles.active=jdbc` (h2:mem). 영속 파일은
  `--spring.datasource.url=jdbc:h2:file:./data/baedal --spring.ai.chat.memory.repository.jdbc.initialize-schema=always`
- 사전 조건: Ollama에 `qwen2.5` 모델 (`ollama list`로 확인).
