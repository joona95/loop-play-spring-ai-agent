## [A] JDBC(h2:mem) 시나리오 1 — 메모리 동작(저장소만 교체, 동작 동일)
  T1 '2024-1234 어디쯤' / T2 '그거 언제 도착'
  → /messages 4건(JDBC 저장소에서 읽음), /ids=["jmem"], T2가 1234 맥락 해소
  (단 일부 응답에 raw tool-call JSON 누출 — qwen2.5 protocol 파손, 메모리/JDBC와 무관한 R2 이슈)

## [B] 재시작 영속성
  InMemory(기본 !jdbc) : ❌ 프로세스 종료 시 소멸(메모리)
  jdbc:h2:mem          : ❌ 재시작 후 jmem /messages=[], /ids=[] (JVM 종료 시 소멸)
  jdbc:h2:file         : ✅ 재시작 후 /ids=["jfile"], /messages 4건 복원,
                            '그 주문 어디쯤이었죠?' → 파일에서 복원된 1234 맥락(7:54:43)으로 응답

################### [A3] h2:file 영속성 + 5종 SELECT — WRITE_DELAY 발견 ###################
# 핵심 발견: h2:file 기본 flush(비동기 WRITE_DELAY≈500ms)는 pkill 시 적재분 유실(2회 재현).
#   WRITE_DELAY=0(동기 flush)으로 재시작 후 복원 성공 — 영속은 'JDBC면 끝'이 아니라 flush 정책에 달림.
#
# Phase1 적재(h2:file;WRITE_DELAY=0): 5종 6세션 /ids=[js1,js2,js3,js4A,js4B,js5]
